# AGENTS.md

## Общие правила работы

- Работай пошагово.
- Не реализуй несколько крупных этапов сразу без явного запроса.
- Перед большими изменениями сначала предложи краткий план.
- Изменяй только необходимые для текущего шага файлы.
- Не делай лишних рефакторингов без запроса.
- Не изменяй несвязанные части проекта, если это не требуется для текущей задачи.

## Ограничения

- Не меняй настройки ESLint, Prettier, TypeScript, Vite и другие конфиги без необходимости.
- Если для задачи требуется изменить конфиги или структуру проекта, сначала объясни зачем.
- Не добавляй новые зависимости без явной необходимости.
- Не откатывай и не переписывай вручную внесённые изменения пользователя без явного запроса.

## Кодстайл

- Используй TypeScript.
- Предпочитай понятные имена переменных и функций.
- Не усложняй архитектуру раньше времени.
- Держи код простым и читаемым.
- Добавляй комментарии только там, где нужно объяснить причину, а не очевидные действия.
- По возможности предпочитай простые функции и понятные структуры данных вместо преждевременного усложнения.

## Формат работы

- После изменений перечисляй изменённые файлы.
- Кратко объясняй, что было сделано.
- Если есть сомнения в архитектурном решении, сначала предложи варианты, а не меняй код сразу.
- По возможности предлагай минимальные изменения, а не широкие переработки.
- Если задача сформулирована неоднозначно, сначала кратко обозначь допущения, на которых основано решение.

## Работа с заметками

- Используй NOTES.md как основной файл с рабочими заметками по проекту.
- В NOTES.md фиксируй текущий план, выполненные шаги, важные решения и следующие задачи.
- Не перезаписывай NOTES.md целиком без необходимости.
- Обновляй только релевантные разделы NOTES.md, сохраняя уже зафиксированный полезный контекст.
- Обновляй NOTES.md после завершения значимого этапа работы, а не после каждого мелкого изменения.
- Не обновляй NOTES.md для мелких локальных фиксов, если они не меняют план, архитектурные решения или текущее состояние этапа.

## Проверка результата

- После реализации шага указывай, что именно стоит проверить вручную.
- Если код не был проверен запуском, явно сообщай об этом.
- Предложи краткое сообщение для коммита в репозиторий на русском и английском.
